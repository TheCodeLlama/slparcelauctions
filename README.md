# SLParcels

Player-to-player land auction platform for Second Life. Bridges a web-based auction UI to in-world Second Life via verification terminals, escrow objects, and bot services. Phase 1 supports Mainland parcels only.

The full design lives in [`docs/initial-design/DESIGN.md`](docs/initial-design/DESIGN.md). Implementation phases and tasks are under [`docs/implementation/`](docs/implementation/).

## Stack

| Layer    | Tech                                                          |
|----------|---------------------------------------------------------------|
| Frontend | Next.js 16, React 19, TypeScript 5, Tailwind 4                |
| Backend  | Spring Boot 4, Java 26, Maven, JPA (Hibernate DDL)            |
| Bot      | .NET 8, LibreMetaverse (see [`bot/README.md`](bot/README.md)) |
| Storage  | PostgreSQL 17, Redis 7, MinIO (S3-compatible)                 |
| In-world | LSL scripts (Phase 6+)                                        |

## Quick start (Docker Compose)

The fastest way to bring up the full stack is with Docker Compose. PostgreSQL, Redis, the Spring Boot backend, and the Next.js frontend all run in containers on a shared network.

```bash
cp .env.example .env       # adjust values if you want non-default ports/credentials
docker compose up --build
```

`.env` is gitignored. Never commit it.

Once everything is healthy:

| Service           | URL                                                        |
|-------------------|------------------------------------------------------------|
| Frontend          | http://localhost:3000                                      |
| Backend API       | http://localhost:8080                                      |
| Backend health    | http://localhost:8080/api/v1/health                        |
| Bot worker health | http://localhost:8081/health (`bot-1`) — `bot-2` defaults to `:8082`. Override via `BOT1_HEALTH_PORT` / `BOT2_HEALTH_PORT` in `.env`. |
| MinIO S3 API      | http://localhost:9000                                      |
| MinIO console     | http://localhost:9001 (`slpa-dev-key` / `slpa-dev-secret`) |
| PostgreSQL        | `localhost:5432` (user `slpa`)                             |
| Redis             | `localhost:6379`                                           |

To stop the stack: `docker compose down`. To reset the database too: `docker compose down -v` (drops the named volumes).

### Port conflicts

The compose stack binds host ports `5432`, `6379`, `8080`, and `3000`. If you already run standalone `slpa-postgres` / `slpa-redis` containers (e.g. from earlier in development), stop them first:

```bash
docker stop slpa-postgres slpa-redis
```

Or override the host ports in `.env`:

```bash
POSTGRES_PORT=5433
REDIS_PORT=6380
```

The container-to-container connections always use the standard ports inside the `slpa-net` network — only the host port mapping changes.

### Hot reload

- **Frontend** — Next.js dev server runs inside the container with the source bind-mounted from `./frontend`. Edits to `.tsx` / `.ts` / CSS files trigger HMR. `WATCHPACK_POLLING=true` is set so file watches survive the WSL2 boundary on Windows hosts.
- **Backend** — `./mvnw spring-boot:run` runs inside the container with `./backend/src` bind-mounted. Source edits do not auto-reload; restart the backend container to pick them up:

  ```bash
  docker compose restart backend
  ```

  Maven dependencies are cached in a named volume (`maven-cache`) so restarts stay fast.

## Local development without Docker

If you'd rather run the backend or frontend on the host (faster JVM start, IDE debugging, etc.) you only need PostgreSQL + Redis from compose:

```bash
docker compose up -d postgres redis
```

Then in two separate shells:

```bash
# Backend
cd backend
./mvnw spring-boot:run                # uses application.yml; pass -Dspring-boot.run.profiles=dev for the dev profile

# Frontend
cd frontend
npm install
npm run dev
```

### Hybrid: host backend + containerised bot

A common dev mode is running the backend in your IDE (debugger attached) while the bot runs in Docker (real .NET 8 + LibreMetaverse, networking already wired). Two snags: `bot-1` declares `depends_on: backend` (so plain `docker compose up bot-1` collides with the IDE-run backend on host port 8080), and the bot's `Backend__BaseUrl` is hardcoded to the in-network `http://backend:8080`. Both fix with a local `compose.override.yml` (gitignored) plus `--no-deps`:

```yaml
# compose.override.yml — local-dev only, do not commit
services:
  bot-1:
    environment:
      Backend__BaseUrl: http://host.docker.internal:8080
```

```bash
docker compose up -d --no-deps bot-1
```

Full walkthrough (incl. Linux `extra_hosts` note, `bot/.dockerignore` rationale, and the bot bring-up smoke flow) lives in [`FULL_TESTING_PROCEDURES.md`](FULL_TESTING_PROCEDURES.md) §3.4–§3.5.

The backend requires `JWT_SECRET` in production (environment variable, ≥ 256 bits base64-encoded) and uses a committed dev default in `application-dev.yml`. The `auth/` slice provides `/api/v1/auth/register`, `/api/v1/auth/login`, `/api/v1/auth/refresh`, `/api/v1/auth/logout`, and `/api/v1/auth/logout-all`. Access tokens are 15-minute HS256 JWTs returned in the response body; refresh tokens are DB-backed, rotated on every refresh, with reuse-detection cascade, held in an HttpOnly `Path=/api/v1/auth` cookie. See [`docs/superpowers/specs/2026-04-11-task-01-07-jwt-auth-backend-design.md`](docs/superpowers/specs/2026-04-11-task-01-07-jwt-auth-backend-design.md) for the full design.

The `verification/` slice provides `GET /api/v1/verification/active` and `POST /api/v1/verification/generate` for the player-verification flow. Codes are 6-digit numerics with a 15-minute TTL; generating a fresh code voids any prior active codes. The handler maps `AlreadyVerifiedException` (409), `CodeNotFoundException` (404), and `CodeCollisionException` (409) to RFC 9457 ProblemDetail responses. See [`docs/superpowers/specs/2026-04-13-epic-02-sub-1-verification-backend.md`](docs/superpowers/specs/2026-04-13-epic-02-sub-1-verification-backend.md).

The `sl/` slice exposes `POST /api/v1/sl/verify`, the header-gated endpoint that an in-world LSL terminal posts to once the player enters their 6-digit code on the device. The path is `permitAll` in `SecurityConfig` because browser JWTs do not exist in-world; `SlHeaderValidator` is the actual trust boundary, checking `X-SecondLife-Shard` against the configured grid and `X-SecondLife-Owner-Key` against `slpa.sl.trusted-owner-keys`. The orchestrator pre-checks `users.sl_avatar_uuid` for a friendly 409 before consuming the verification code, then links the avatar to the user account and marks `verified=true`. Slice-scoped `SlExceptionHandler` overrides the verification-package responses (400 for not-found, 409 for collision, 409 for the constraint-race avatar-already-linked path via `ConstraintNameExtractor`). The `slpa.sl` config block lives in `application.yml` (`expected-shard: Production`, empty key list), `application-dev.yml` (placeholder UUID `00000000-0000-0000-0000-000000000001` for Postman + integration tests), and `application-prod.yml` (empty list — the deploy pipeline injects real UUIDs and `SlStartupValidator` fails fast on `prod` startup if the list is still empty).

The `sl/` slice also holds two WebFlux `WebClient` clients used by the parcel lookup flow in Epic 03. `SlWorldApiClient` fetches parcel metadata HTML from `world.secondlife.com/place/{uuid}` and parses it with Jsoup against the meta tags SL actually serves today: `name=parcel` (title), `name=region`, `name=ownerid` (null for `ownertype=group`), `name=ownertype`, `name=area`, `name=location` (slash-separated `x/y/z` parcel-local coords), plus body selectors for `<p class="desc">` and `<img class="parcelimg">` (snapshot URL, with `name=snapshot` taking precedence when SL populates it). Maturity rating is **not** read from the parcel page — SL scopes maturity to the region, and the parcel page does not reliably expose it; the field stays null at ingest until user input lands at listing-creation time. The client retries 5xx / network errors with exponential backoff (`slpa.world-api.retry-attempts`, `slpa.world-api.retry-backoff-ms`), fails fast on 404 with `ParcelNotFoundInSlException` (mapped before `bodyToMono` so the retry loop does not eat it), and wraps any other failure as `ExternalApiTimeoutException`; if a future SL HTML drift breaks parsing, the resulting `ParcelIngestException` surfaces as 422 `PARCEL_INGEST_FAILED` rather than a 500. `SlMapApiClient` POSTs to the SL Map CAP endpoint at `cap.secondlife.com/cap/0/{cap-uuid}` with a `var=loc&sim_name=<regionName>` form body (the `var` value is an arbitrary JS identifier echoed in the response) and parses the `var loc = {'x' : N, 'y' : N };` JavaScript response via regex; it runs the same symmetric retry loop as `SlWorldApiClient` (`slpa.map-api.retry-attempts`, `slpa.map-api.retry-backoff-ms`) and throws `RegionNotFoundException` (→ HTTP 404) when the Map API returns 200 with no `'x'`/`'y'` literal (i.e. the region name does not exist), distinct from the generic `ExternalApiTimeoutException` (→ HTTP 504) used for actual gateway failures. `MainlandContinents` is a static helper holding the 17 continent bounding boxes from the SL wiki's `ContinentDetector` page — it replaces the unreliable Grid Survey API dependency with a half-open-interval point-in-box check. Both clients are covered by WireMock-backed unit tests; Jsoup 1.17.2 and `org.wiremock:wiremock-standalone:3.10.0` (test scope) are the new deps.

The `parcel/` slice exposes `POST /api/v1/parcels/lookup` — an authenticated + SL-verified endpoint that resolves a parcel UUID to a transient `ParcelLookupResult` (no shared persistence; per-auction snapshots live on `AuctionParcelSnapshot`). `ParcelLookupService` fetches World API metadata, resolves region → grid coords via the Map API, runs the Mainland continent check, and synthesizes a SLURL (`https://maps.secondlife.com/secondlife/{region}/{x}/{y}/{z}`, defaults `128/128/22` if the World API didn't return a position). Verification gating is inline in `ParcelController`: Spring Security enforces the JWT auth (401), then the controller throws `AccessDeniedException` (→ 403 via `GlobalExceptionHandler`) when `user.verified != true`. The five SL-flow exceptions have mappings on `GlobalExceptionHandler`: `ParcelNotFoundInSlException` → 404 / `SL_PARCEL_NOT_FOUND`, `RegionNotFoundException` → 404 / `SL_REGION_NOT_FOUND`, `NotMainlandException` → 422 / `NOT_MAINLAND`, `ParcelIngestException` → 422 / `PARCEL_INGEST_FAILED` (parser drift defense in depth), `ExternalApiTimeoutException` → 504 / `SL_API_TIMEOUT`. `ParcelLookupServiceTest` covers the 4-case mocked matrix (new UUID on Mainland, cache hit, World API 404, non-Mainland coords); `ParcelControllerIntegrationTest` covers the 6-case endpoint matrix (happy path, cache hit verified via `Mockito.verify(worldApi, times(1))`, non-Mainland 422, malformed UUID 400, unauthenticated 401, unverified 403) under a full Spring Boot context with `@MockitoBean` on the SL clients so no outbound network calls happen.

The `auction/` slice adds auction CRUD + state machine + cancellation (Epic 03 sub-spec 1, Task 4). `POST /api/v1/auctions`, `GET /api/v1/auctions/{publicId}`, `GET /api/v1/users/me/auctions`, `PUT /api/v1/auctions/{publicId}`, `PUT /api/v1/auctions/{publicId}/verify`, `PUT /api/v1/auctions/{publicId}/cancel`, and `GET /api/v1/auctions/{publicId}/preview` round out the surface. Write paths are SL-verified (same inline guard as the parcel slice); reads require auth but not verification. The linchpin is the **two-DTO model with type-enforced status collapse**: `SellerAuctionResponse` carries the full internal state (commission, listing fee, winnerId, verification notes, pendingVerification), while `PublicAuctionResponse` is typed on `PublicAuctionStatus { ACTIVE, ENDED }` — the four terminal statuses (COMPLETED / CANCELLED / EXPIRED / DISPUTED) collapse to `ENDED` in an exhaustive `switch`, and the four pre-ACTIVE statuses (DRAFT / DRAFT_PAID / VERIFICATION_PENDING / VERIFICATION_FAILED) throw `IllegalStateException` from the mapper because `AuctionController.get()` is responsible for 404-hiding those from non-sellers before the mapper is ever called. Field-level validation is layered: JSR-380 (`@NotNull`, `@Min`, `@Size(max=10)` on tags, `@Size(max=5000)` on sellerDesc) at the DTO, plus `AuctionService` checks for the enum sets (`durationHours ∈ {24,48,72,168,336}`, `snipeWindowMin ∈ {5,10,15,30,60}` iff `snipeProtect`) and the pricing monotonicity (`reserve >= starting`, `buyNow >= max(starting, reserve ?? 0)`). `CancellationService` implements the state matrix: DRAFT / DRAFT_PAID / VERIFICATION_PENDING / VERIFICATION_FAILED / ACTIVE (pre-`endsAt`) are cancellable, everything else throws `InvalidAuctionStateException` (→ 409 via the slice-scoped `AuctionExceptionHandler`); a `CancellationLog` row is always written, a `ListingFeeRefund` row (PENDING) is written only when `listingFeePaid && from != ACTIVE`, and `user.cancelledWithBids` increments only on ACTIVE+bids cancellations. `AuctionExceptionHandler` sits at `@Order(HIGHEST_PRECEDENCE + 10)` and maps `AuctionNotFoundException` → 404, `InvalidAuctionStateException` → 409, and `IllegalArgumentException` (from the service-layer validation + unknown tag codes + missing parcel) → 400. Test coverage: `AuctionDtoMapperTest` (17 cases — parameterized status-collapse matrices + public-view field hiding), `AuctionServiceTest` (19 cases — pricing / duration / snipe / tag resolution / update-state matrix), `CancellationServiceTest` (14 cases — state × fee × bids matrix + cancel-after-end), `AuctionControllerIntegrationTest` (14 cases — create happy path, unverified 403, bad parcelId 400, seller-vs-public-view split, 404-hides-pre-ACTIVE, cancel refund-row round-trip, cancel-from-COMPLETED 409). `slpa.listing-fee.amount-lindens` (default 100) and `slpa.commission.default-rate` (default 0.05) are new in `application.yml`.

The `Users/` folder holds `Create user`, `Get current user`, `Get user by id`, `Update current user` (PUT /me), `Upload avatar` (multipart — pick a local file from the Body tab before firing), and `Get user avatar` (tweak `avatarSize` to `64` / `128` / `256` to flip sizes). `Tags/` holds `List`, and `Photos/` holds `Upload` (multipart), `Delete`, and `Get bytes` (public, no auth). The collection is the canonical manual-test surface for the backend; if you add a new endpoint, add it to the collection in the same task.

`DevAuctionController` (Task 5 of Epic 03 sub-spec 1) adds a second dev-profile-only helper at `POST /api/v1/dev/auctions/{id}/pay` that stands in for the real in-world listing-fee payment terminal callback arriving in Epic 05. Body is an optional `{ amount?: Long, txnRef?: string }`; amount defaults to `slpa.listing-fee.amount-lindens` (100) and txnRef defaults to `"dev-mock-<uuid>"`. Transitions DRAFT → DRAFT_PAID and stamps `listingFeePaid=true, listingFeeAmt, listingFeeTxn, listingFeePaidAt=now(clock)`; the `AuctionService.loadForSeller` 404s non-sellers, and any status other than DRAFT throws `InvalidAuctionStateException` (→ 409). The same three-layer gating applies: `@Profile("dev")` is the real trust boundary, `SecurityConfig` permits `/api/v1/dev/**`, and the controller explicitly rejects null principals with `AccessDeniedException` (→ 403 `ACCESS_DENIED`) because the permitAll path skips the normal JWT chain's 401 handling. `DevAuctionControllerTest` is a 6-case `@SpringBootTest` covering pay-on-DRAFT (defaults persist), pay-on-DRAFT_PAID (409), pay-on-ACTIVE (409), pay-as-non-seller (404), pay-with-custom-body (exact values persist), and pay-without-auth (403).

**Verification flow (current, since spec `2026-05-16-ownership-only-verification-design`).** Verification is a single synchronous SL World API ownership check at listing activation time. `PUT /api/v1/auctions/{publicId}/verify` takes no body. The backend compares the World-API-reported parcel owner UUID to the expected UUID (seller's avatar for individual listings, registered SL group for case-3 group land) and transitions DRAFT_PAID → ACTIVE on match or DRAFT_PAID → VERIFICATION_FAILED on mismatch. No VERIFICATION_PENDING intermediate state, no bot involvement, no in-world setup. ACTIVE-state monitoring continues via the World-API-driven `OwnershipCheckTask`, gated by a streak threshold (`slpa.ownership-monitor.mismatch-streak-threshold`, default 2) so a single transient SL API blip cannot suspend a live auction. The three Epic 03 sub-spec 1 historical entries below (Tasks 6 / 7 / 8) describe the retired Method A / B / C dispatcher; their endpoints (`POST /api/v1/sl/parcel/verify`, `POST /api/v1/bot/tasks/{id}/monitor`, `POST /api/v1/dev/bot/tasks/{id}/complete`) and the `VerificationMethod` enum, `PendingVerification` DTO, and bot `MONITOR_AUCTION` / `MONITOR_ESCROW` / `VERIFY` task types are all gone. The bot's only remaining role is `WITHDRAW_GROUP` task handling plus SL IM dispatch and idle parking.

`AuctionVerificationService` (Task 6 of Epic 03 sub-spec 1) wires `PUT /api/v1/auctions/{publicId}/verify` as the unified verify entry point. `triggerVerification(auctionId, sellerId)` loads via `AuctionService.loadForSeller` (404 on non-seller), rejects any source state outside `{DRAFT_PAID, VERIFICATION_FAILED}` with `InvalidAuctionStateException` (→ 409), flips to VERIFICATION_PENDING clearing `verification_notes`, then dispatches by `verificationMethod`. Method A (UUID_ENTRY) runs inline: calls the World API via `SlWorldApiClient`, catches any `RuntimeException` or group-owner or UUID-mismatch to record a failure note + transition to VERIFICATION_FAILED, otherwise runs the service-layer parcel-lock pre-check (`assertParcelNotLocked` via `existsByParcelIdAndStatusInAndIdNot` + `findFirstByParcelIdAndStatusIn`) which throws `ParcelAlreadyListedException(parcelId, blockingAuctionId)`, refreshes `parcel.ownerUuid/ownerType/lastChecked` from the fresh World API data, then transitions to ACTIVE with `startsAt=now, endsAt=now+durationHours, originalEndsAt=endsAt, verifiedAt=now, verificationTier=SCRIPT`. The final `auctionRepo.saveAndFlush` forces the INSERT/UPDATE to hit Postgres inside the try/catch so the **partial unique index backstop** (`uq_auctions_parcel_locked_status`, created at boot by `ParcelLockingIndexInitializer`) surfaces as `DataIntegrityViolationException` → `ParcelAlreadyListedException(parcelId, -1L)` (the winning transaction's auction ID is unavailable at catch-time; -1 is the sentinel for "race-caught"). `AuctionExceptionHandler` adds `ParcelAlreadyListedException` → 409 with `code=PARCEL_ALREADY_LISTED`, `parcelId`, `blockingAuctionId`. Tests: `AuctionVerificationServiceMethodATest` (11 mocked-repo cases covering happy path from DRAFT_PAID + retry from VERIFICATION_FAILED, 409 source states, group-owner / UUID-mismatch / World API timeout failure paths, service-layer lock conflict, DB unique-index backstop with sentinel) and `ParcelLockingRaceIntegrationTest` (3 full-stack Postgres cases: sequential verify where A1 wins and A2 gets 409 identifying A1 as blocker, cancel-unblocks-retry where A2 can verify after A1 is cancelled, and a `CompletableFuture`-driven concurrent race where exactly one of two parallel verify calls wins 200 and the other gets 409 from the DB backstop).

Task 8 wires Method C (SALE_TO_BOT): `dispatchMethodC` calls `BotTaskService.createForAuction(auction)` which first marks any prior PENDING/IN_PROGRESS bot task for that auction FAILED with reason `"Superseded by retry"` (so retries from VERIFICATION_FAILED cleanly rotate tasks), then persists a fresh `bot_tasks` row (type=VERIFY, status=PENDING, `parcel_uuid`, `region_name`, `sentinel_price`=`slpa.bot-task.sentinel-price-lindens`). The auction stays in VERIFICATION_PENDING. `buildPendingVerification` for SALE_TO_BOT queries the latest open bot task (PENDING first, IN_PROGRESS fallback) and populates `SellerAuctionResponse.pendingVerification = { method: SALE_TO_BOT, botTaskId, instructions }` with a human-readable string `"Set your parcel for sale to SLPAEscrow Resident (UUID: <primary escrow UUID>) at L$999,999,999. A verification worker will confirm within 48 hours."`. `BotTaskController` exposes `GET /api/v1/bot/tasks/pending` (FIFO queue for the bot worker), `PUT /api/v1/bot/tasks/{taskId}` (worker callback), and the new atomic `POST /api/v1/bot/tasks/claim` from Epic 06 Task 2 — all three are now bearer-token gated as of Epic 06 Task 3 (`.requestMatchers("/api/v1/bot/**").access(...)` in `SecurityConfig` consults `BotSharedSecretAuthorizer.isAuthorized`, which parses the `Authorization: Bearer <secret>` header and compares against `slpa.bot.shared-secret` using `MessageDigest.isEqual` for a constant-time compare that does not leak length via timing). `BotStartupValidator` (`@Profile("!dev")`) fails fast on non-dev profiles if the secret is blank, equal to the dev placeholder `"dev-bot-shared-secret"`, or shorter than 16 characters, and also refuses the dev placeholder `slpa.bot-task.primary-escrow-uuid` (`00000000-0000-0000-0000-000000000099`). Config lives on the new `BotTaskConfigProperties` record (`@ConfigurationProperties(prefix = "slpa")`, registered by `BotPropertiesConfig`) with two nested records (`Bot` + `BotTask`) so the authorizer and validator share a single typed read — existing `@Value` reads against `slpa.bot-task.*` keys coexist on the same YAML tree and will migrate to the record in Task 12. `BotTaskService.complete` drives the terminal transition: on `SUCCESS` it validates `body.authBuyerId == slpa.bot-task.primary-escrow-uuid` and `body.salePrice == sentinelPrice` (else 400 `INVALID_REQUEST`), runs the same `existsByParcelIdAndStatusInAndIdNot` parcel-lock pre-check as Methods A/B (on conflict: marks the task FAILED with reason `PARCEL_LOCKED`, flips the auction to VERIFICATION_FAILED, throws `ParcelAlreadyListedException` identifying the blocker), refreshes parcel metadata (owner/area/region/position/lastChecked) from the body, stamps `result_data` as a JSON map, and transitions the auction to ACTIVE with `verificationTier=BOT` via `saveAndFlush` (DB unique-index backstop → `ParcelAlreadyListedException(-1L)` on concurrent races). On `FAILURE` it marks the task FAILED + flips the auction to VERIFICATION_FAILED with `verification_notes = "Bot: <reason>"` — no refund (seller can retry via `PUT /verify`). `DevBotTaskController` (`@Profile("dev")`) exposes `POST /api/v1/dev/bot/tasks/{id}/complete` with identical body and behavior for browser-driven testing. `BotTaskTimeoutJob` is a `@Scheduled(fixedDelayString = "${slpa.bot-task.timeout-check-interval:PT15M}")` sweep that calls `BotTaskService.findPendingOlderThan(Duration.ofHours(timeoutHours))` and `markTimedOut` on each stale task (task FAILED with reason `TIMEOUT`, auction flipped to VERIFICATION_FAILED only if still VERIFICATION_PENDING as a defensive guard for cancelled auctions). `BotTaskExceptionHandler` sits at `@Order(HIGHEST_PRECEDENCE + 10)` scoped to `com.slparcelauctions.backend.bot` and maps `IllegalArgumentException` → 400 `INVALID_REQUEST`, `InvalidAuctionStateException` → 409 `AUCTION_INVALID_STATE`, and `ParcelAlreadyListedException` → 409 `PARCEL_ALREADY_LISTED` (needed because `AuctionExceptionHandler` is base-package-scoped to `...auction` and does not see exceptions raised from the bot package). `application.yml` adds the `slpa.bot-task.*` block (`sentinel-price-lindens: 999999999`, `primary-escrow-uuid: 00000000-0000-0000-0000-000000000099` dev placeholder, `timeout-hours: 48`, `timeout-check-interval: PT15M`); prod overrides `primary-escrow-uuid` via env var. ### Realty Groups

Brokerage-style groups (`realty/` slice) let a leader manage a roster of agents, where each `(group, agent)` pair carries a permission flag set gating administrative actions on that group. A user can belong to many groups simultaneously, including leading more than one. The slice ships sub-projects A (Realty Group core) and B (Permissions model); C (group-listing integration, issue #237), D (group wallet, #238), E (SL group correlation, #239), and F (admin moderation, #240) load on top in later cycles. Design lives in [`docs/superpowers/specs/2026-05-10-realty-groups-core-permissions-design.md`](docs/superpowers/specs/2026-05-10-realty-groups-core-permissions-design.md).

- **Roles.** Each group has exactly one leader (a transferable seat) plus N agents. Leadership transfer takes a body `{ newLeaderPublicId, oldLeaderAction: "STAY" | "LEAVE" }` so the outgoing leader can stick around as an agent or leave outright. Leader cannot leave; must transfer first (409 `LEADER_CANNOT_LEAVE`).
- **Permissions.** Per-(group, agent) `TEXT[]` flag set persisted as `Set<RealtyGroupPermission>` via Hibernate's `@JdbcTypeCode(SqlTypes.ARRAY)`. Four flags ship in A+B: `INVITE_AGENTS`, `REMOVE_AGENTS`, `EDIT_GROUP_PROFILE`, `CONFIGURE_FEES`. The leader has all of them implicitly — the membership row's `permissions` column is meaningless for the leader. `RealtyGroupAuthorizer` is the per-(user, group, permission) check used by every mutating endpoint.
- **Public profile.** Each group has a public page at `/group/{slug}` rendering name + logo + cover + description + leader card + public agent roster + fee rates. Slug auto-derives from the name on create and regenerates on rename. Renames are gated by a 30-day cooldown for non-admins (409 `GROUP_RENAME_COOLDOWN` carries `cooldownEndsAt`); admins bypass via the force-edit endpoint.
- **Dissolution.** `DELETE /api/v1/realty-groups/{publicId}` (leader only) is a soft delete — sets `dissolved_at`. The group disappears from public + member listings. Because the `name_lower` and `slug` unique indexes are partial on `dissolved_at IS NULL`, the name + slug become immediately reusable by a fresh group.
- **Invitations.** Group leaders + agents with `INVITE_AGENTS` invite by username with a chosen permission set. A `RealtyGroupInvitationExpiryJob` (Spring `@Scheduled`) flips invitations to `EXPIRED` after their TTL (`slpa.realty.invitation-ttl-days`, default 7). The partial unique `(group_id, invited_user_id) WHERE status = 'PENDING'` index blocks duplicate live invitations (409 `INVITATION_ALREADY_PENDING`), and `member_seat_limit` (default `slpa.realty.default-member-seat-limit`, 50) caps invite issuance (409 `SEAT_LIMIT_REACHED`).
- **Admin escape hatches.** ROLE_ADMIN endpoints at `/api/v1/admin/realty-groups` cover paginated list (filterable by `status=active|dissolved`, `leader`, `search`), single read (including dissolved), force-edit (bypasses the 30-day rename cooldown), force-dissolve (no leader gate), and force-remove-member with an optional `?newLeaderPublicId=<userPublicId>` query parameter when removing the leader. All five writes thread `AdminActionService` with new `AdminActionType` values: `REALTY_GROUP_EDIT`, `REALTY_GROUP_DISSOLVE`, `REALTY_GROUP_MEMBER_REMOVE`. Frontend pages at `/admin/realty-groups` (list) and `/admin/realty-groups/[publicId]` (detail).
- **Cross-cutting image pipeline.** Group logo + cover uploads go through a new central `ImageStorageService` chokepoint that converts every raster upload (JPG / PNG / WebP / BMP) to WebP at write time. Avatars, listing photos, and default covers all migrated to the same helper in the same epic, so the entire S3 image surface is WebP now.

### Realty Groups: Group Wallet (Sub-project D)

Adds a balance-bearing wallet to each realty group and completes the agent-fee L$ distribution deferred by C.

- **New endpoints:** `GET /api/v1/realty/groups/{publicId}/wallet` (balance + 50-entry recent ledger), `GET .../wallet/ledger?cursor={createdAt}&limit=50` (cursor-paginated full history), `POST .../wallet/withdraw` (initiate leader-avatar withdrawal, idempotent via client-supplied UUID key). Permission gates: `VIEW_GROUP_TRANSACTIONS` on reads, `WITHDRAW_FROM_GROUP_WALLET` on the withdraw. Leader holds all permissions implicitly.
- **Listing-fee routing:** Group-listed auctions debit the listing fee from the group wallet at `pay-listing-fee` time. `MeWalletController.payListingFee` branches on `auction.realty_group_id`; individual-listing debits are unchanged.
- **Agent-fee distribution at escrow-completion:** `floor(agent_fee_amt × agent_fee_split)` credits the group wallet (ledger type `AGENT_FEE_CREDIT`); the remainder credits the listing agent's user wallet. Distribution runs inside the `handleEscrowPayoutSuccess` transaction boundary via `AgentFeeDistributor`, so a rollback leaves both ledgers and the payout audit rows consistent.
- **Listing-fee refund routing:** `ListingFeeRefundProcessor` looks up the originating `realty_group_ledger.LISTING_FEE_DEBIT` row. If found, the refund credits the group wallet; otherwise it falls through to the user wallet. Correctly handles C-era group listings (fee was user-debited) and D-era group listings (fee was group-debited) without a denorm column.
- **Dissolution gate tightening:** `RealtyGroupService.dissolve` now requires `balance_lindens = 0`, `reserved_lindens = 0`, and no in-flight escrows (`FUNDED`, `AWAITING_TRANSFER`, `DISPUTED`) in addition to C's "no active listings" check. New exceptions: `GroupHasNonzeroBalanceException` (409), `GroupHasInFlightEscrowsException` (409).
- **Group dormancy:** Weekly `GroupWalletDormancyJob` (4:30 AM UTC Mondays, 30 min after the user-wallet sweep). Active signal: any current member's refresh-token rotation within 30 days. Four weekly IMs to the group leader (phases 1–4); phase 4 triggers auto-return of the full balance to the leader's SL avatar via a `TerminalCommand`. Any balance-changing write path clears the dormancy phase as defense-in-depth. Member login or token rotation also clears the phase for all groups they belong to.
- **User wallet dormancy** (parallel job): Weekly `UserWalletDormancyJob` (4:00 AM UTC Mondays, 30 min before the group sweep). Active signal: the user's own refresh-token rotation within 30 days. Same four-phase escalation; phase 4 queues a `TerminalCommand{WITHDRAW, USER_WALLET_DORMANCY_AUTO_RETURN}` to the user's SL avatar for the *available* balance only (`balance_lindens - reserved_lindens`), leaving any live bid reservations intact. Login / refresh-token rotation clears phases 1–4 (phase 99 is preserved). New `UserLedgerEntryType.DORMANCY_AUTO_RETURN`, Flyway migration V40. Spec at `docs/superpowers/specs/2026-05-19-user-wallet-dormancy-design.md`.
- **New permission enum values:** `WITHDRAW_FROM_GROUP_WALLET` (wired to POST withdraw), `VIEW_GROUP_TRANSACTIONS` (wired to GET wallet + ledger). `VIEW_GROUP_TRANSACTIONS = true` is in the default-invite permission set; `WITHDRAW_FROM_GROUP_WALLET = false` is opt-in.
- **Flyway migration V26** adds wallet columns to `realty_groups`, creates the `realty_group_ledger` append-only table, adds `realty_group_id` to `terminal_commands` (for completion-handler ledger routing), extends `reconciliation_runs` with `group_wallet_balance_total` / `group_wallet_reserved_total`, and widens the `user_ledger_entry_type_check` constraint to include `AGENT_FEE_CREDIT`.
- **Frontend:** `/realty/groups/[publicId]/wallet` (client-only, `dynamic = "force-dynamic"`). Components: `GroupWalletBalanceCard`, `GroupWalletLedgerTable` (cursor-paginated, "load more"), `GroupWithdrawModal` (client-side idempotency-key generation), `LeaderTermsBlockBanner`. `AgentFeePreview` in the listing wizard extends to show a wallet-source line ("Listing fee paid from [Group] wallet — current balance L$X") and, when balance is insufficient, a destructive shortfall message that disables the publish button.

Spec at `docs/superpowers/specs/2026-05-11-realty-groups-group-wallet-design.md`.

### Coupon codes

Admin-managed promotional codes that discount listing fees and/or commission rates, with per-user redemption (wallet screen, create-listing screen), allowlist gating, signup-window auto-grant, and full admin CRUD UI.

- **Templates + grants.** `coupons` carries the redemption key, lifetime axes (`duration_days` and/or `use_count`), redemption controls (`redeemable_until`, `max_total_redemptions`, `max_per_user`), optional `signup_window_start/end`, and an `active` toggle. `coupon_discounts` holds 1:N discount lines (`target` LISTING_FEE | COMMISSION_RATE, `op` OVERRIDE | PERCENT_OFF | FLAT_OFF, `value`). `coupon_allowed_users` enforces a per-coupon allowlist. `coupon_grants` is the per-user instance carrying `expires_at`, `remaining_count`, `state` ACTIVE | EXHAUSTED | EXPIRED | REVOKED, and `source` REDEMPTION | ADMIN_GRANT | SIGNUP_WINDOW. Auctions snapshot `listing_fee_coupon_grant_id` + `commission_coupon_grant_id` at create time (Flyway V41).
- **Resolution and snapshotting.** `CouponDiscountResolver.resolve(userId)` is called once per listing creation, computes the lowest-result line per target across all ACTIVE grants for the user, and stamps the grant ids onto the auction alongside the already-discounted `listingFeeAmt` + `commissionRate`. `ListingFeePaymentService` auto-transitions DRAFT to DRAFT_PAID at L$0 (no terminal hit) when fully discounted. `AuctionActivationService` decrements `remaining_count` for each stamped grant at activation (DRAFT_PAID -> ACTIVE), transitioning to EXHAUSTED at zero; cancellation before ACTIVE does not consume. `EscrowCommissionCalculator` reads the per-auction `commissionRate` snapshot at payout time, so a COMMISSION_RATE discount carries through to the escrow `commissionAmt` / `payoutAmt`.
- **Endpoints.** User-facing: `GET /api/v1/me/coupons` (filter active|history), `POST /api/v1/me/coupons/redeem`, `GET /api/v1/me/listings/prospective-discounts`. Admin: `GET / POST / PATCH / DELETE /api/v1/admin/coupons[/{publicId}]`, `GET / POST /api/v1/admin/coupons/{publicId}/grants`, `POST .../grants/{grantPublicId}/revoke`. All admin endpoints are `ROLE_ADMIN`-gated and emit audit rows.
- **Auto-grant.** Creating a coupon with `signup_window_start/end` runs an in-transaction backfill for all matching existing users. A post-commit hook on `UserService.create()` (`applySignupWindowCoupons`) handles forward users. Both paths share `CouponService.createGrant`, which also fires `NotificationPublisher.couponGranted` (in-app + SL IM via the existing dispatcher) for ADMIN_GRANT and SIGNUP_WINDOW sources; REDEMPTION is silent (the wallet UI already echoed the redeem inline).
- **Sweeper.** `CouponGrantSweeper` runs hourly (`slpa.coupons.sweeper-cron`, default `0 0 * * * *`, UTC) to flip ACTIVE grants whose `expires_at` is past to EXPIRED. The resolver also filters expired-by-clock grants defensively so sweeper lag never produces incorrect snapshots.
- **Frontend.** `WalletCouponsCard` (under the balance card on `/wallet`) handles redeem + active/history lists. `CreateListingCouponSummary` renders inline discount badges on the create-listing summary block plus a "Have a code?" expander that hits `/redeem` inline and refetches the prospective-discounts. Admin pages at `/admin/coupons` (list with status/discount filters), `/admin/coupons/new` (sectioned create form with repeatable discount-bundle rows + allowlist user-picker + signup-window), and `/admin/coupons/{publicId}` (Overview / Grants / Edit tabs with direct-grant modal and revoke confirmation). Sidebar link "Coupons" added to admin nav.

Spec at `docs/superpowers/specs/2026-05-20-coupon-codes-design.md` ([#165](https://github.com/TheCodeLlama/slparcelauctions/issues/165)).

The `onboarding/` slice adds two forced post-verify steps gated by new boolean columns on `users` (`avatar_step_completed`, `display_name_step_completed`, both backfilled in V20 from `profile_pic_url IS NOT NULL` / `display_name IS NOT NULL`). `SlProfilePhotoService` scrapes `world.secondlife.com/resident/{slAvatarUuid}` for the `<img class="parcelimg">` element, allow-lists the `https://picture-service.secondlife.com/` host, fetches the JPEG bytes through a 5s/5s WebClient with a 2MB cap, and Redis-caches results for 1h positive / 5min negative (sentinel `"NONE"`) so a thundering-herd of no-photo users can't hammer LL. `OnboardingController` exposes `GET /api/v1/users/me/onboarding/sl-profile-photo` (proxies bytes with `Cache-Control: private, max-age=3600`, 404 when no SL UUID or no photo), `POST /avatar/skip` (idempotent flag flip), and `POST /display-name` (laxer `@Size(max=50)` only — empty / whitespace / null are skip semantics, non-blank is trimmed and persisted; the stricter `UpdateUserRequest` regex still applies to subsequent settings edits). `AvatarService.upload` also flips `avatar_step_completed` so an upload is one of the three exits from the gate. Frontend `(verified)/(onboarded)/layout.tsx` chains a redirect after the existing verify gate: `!avatarStepCompleted` → `/dashboard/avatar`, `!displayNameStepCompleted` → `/dashboard/display-name`, else children. The display-name page pre-fills the input from `user.username` so the happy path is one click; an empty / cleared submit and the explicit Skip button both POST `null` and trigger the username fallback in `User.getDisplayName()`. Settings page reuses the same `<AvatarCropper>` so onboarding-picked and settings-picked avatars look identical.

The frontend ships the full `/browse` shell (filter sidebar + bottom-sheet on mobile, URL-synced state, SSR-seeded initial render, active-filter chips, Clear all, pagination), three featured rails on the landing page with partial-failure isolation, seller-scoped `/users/[id]/listings`, and the Curator Tray (heart on every ListingCard, right-anchored glass drawer on desktop, BottomSheet on mobile, URL-synced `/saved` page). Every heart toggle uses optimistic updates with rollback on error and targeted toast copy for the three known error codes (409 `SAVED_LIMIT_REACHED`, 403 `CANNOT_SAVE_PRE_ACTIVE`, 404). The SSR initial fetch is resilience-wrapped (`resolveBrowseInitialData` in `src/lib/api/auctions-search.ts`): a backend 4xx filter rejection (e.g. an unknown `near_region` → `REGION_NOT_FOUND`) no longer hard-500s the route; it renders an empty result set with the ProblemDetail code threaded into `BrowseShell` so the sidebar shows an inline message, while a 5xx/network error rethrows into the route-scoped `src/app/browse/error.tsx` boundary (defense-in-depth fallback with a `reset()` "Try again" and a no-filter recovery link). The `near_region` filter is a region-name autocomplete (`DistanceSearchBlock` accessible combobox/listbox): typing debounce-fetches resolvable region names from `GET /api/v1/search/suggest?...&regionsOnly=true` (250ms, min 2 chars, ArrowUp/Down/Enter/Escape, click-select, outside-click close). The filter commits only on an explicit suggestion-select, so every committed value is a verbatim resolvable region name and the search never 400-storms; blurring with un-selected text reverts the input to the last committed region, and clearing it still drops the filter. Distance is a single 0..20 value driven by two synced controls (a native range slider plus an editable, clamped number input) that default to 0 (the selected region only); selecting a region establishes an explicit integer distance (preserving an already-set value, otherwise 0) so the backend never applies its own larger fallback radius, and both controls are disabled with a "pick a region first" helper until a region is set. The SSR-resilience layer and `src/app/browse/error.tsx` boundary remain as the backstop for hand-edited `?near_region=junk` URLs.

## Production deployment

Production runs on AWS in `us-east-1`. Architecture and decisions are documented in [`docs/superpowers/specs/2026-04-29-aws-deployment-design.md`](docs/superpowers/specs/2026-04-29-aws-deployment-design.md). All infrastructure is Terraform-managed under `infra/` (the lone exception is the Amplify app, deliberately console-managed — `infra/main.tf` documents the reason and reproduction steps).

### Live endpoints

- Frontend: `https://slparcels.com` (Amplify Hosting, Next.js SSR on WEB_COMPUTE)
- Backend API: `https://slpa.app` (ALB → ECS Fargate)
- Bots: 5-named-worker pool, currently 2 active (ECS Fargate, no public exposure)

### Auto-deploy (GitHub Actions)

Push to `main` triggers the relevant deploy workflow via OIDC-assumed role (no long-lived AWS keys in repo secrets):

- `backend/**` changed → `.github/workflows/deploy-backend.yml` builds + pushes to ECR + registers a new task-def revision + rolls `slpa-prod-backend` (waits for service stability).
- `bot/**` changed → `.github/workflows/deploy-bot.yml` does the same and fans out across every active `slpa-prod-bot-*` service.
- `frontend/**` changed → Amplify auto-builds from `main`.
- `infra/**` changed → operator runs `terraform plan` / `apply` locally. CI for Terraform is deferred.

Manual one-off deploys via `Actions → deploy backend → Run workflow` (uses the same OIDC role).

### Observability

- **CloudWatch alarms** (11 spec'd, 1 deferred): backend 5xx rate, p95 latency, unhealthy hosts, ECS task failures, RDS CPU / free storage / connections, Redis CPU / memory pressure, per-bot running task count. All publish to the `slpa-prod-alerts` SNS topic; operator email subscription must be confirmed via the AWS-sent email link before alerts arrive.
- **CloudWatch dashboard** at `slpa-prod-overview` — backend traffic + latency, RDS, Redis, ECS task counts.
- **AWS Budgets**: soft $200/mo (actual), hard $400/mo (actual + 80% forecast).
- **Logs**: 7-day retention on every ECS log group; `awslogs-stream-prefix=ecs` for IDE jump-to-stream.

### Cost guardrails

Running cost at launch-lite defaults: ~$78/mo. Upgrade levers in `infra/variables.tf` (Multi-AZ RDS, replicated Redis, larger task sizes, NAT Gateway HA) — each one is a `.tfvars` flip + `terraform apply`.

## Conventions

Read [`docs/implementation/CONVENTIONS.md`](docs/implementation/CONVENTIONS.md) before contributing. Highlights:

- Lombok is required for entities, services, and controllers — no hand-written getters/setters/loggers.
- Schema is managed by Hibernate `ddl-auto: update` during active development. Entity changes do not require a Flyway migration — wipe the DB if a breaking change is needed. Existing migrations in `backend/src/main/resources/db/migration/V<N>__*.sql` run against fresh DBs to apply the baseline. See `CLAUDE.md` for the DB wipe procedure.
- Each task ships as one vertical slice (entity → repo → service → controller → tests).
- Feature-based packages (`user/`, `parcel/`, `auction/`, …), not layer-based.
