# Task 02-02: SL Verification Endpoint

## Goal

Build the backend endpoint that receives verification requests from in-world LSL scripts, validates the SL-injected security headers, and links the SL avatar identity to the user's web account.

## Context

Verification codes are generated by Task 02-01. This endpoint is called by LSL scripts running in Second Life. SL automatically injects tamper-proof HTTP headers on all outgoing requests from scripts - these are our primary identity proof.

## What Needs to Happen

- Create endpoint: POST /api/v1/sl/verify (publicly accessible - no JWT required, since it's called from SL scripts)
- Accept request body with:
  - verification_code (6-digit string)
  - avatar_uuid (UUID)
  - avatar_name (string - legacy name)
  - display_name (string)
  - username (string)
  - born_date (string/date)
  - pay_info (integer - DATA_PAYINFO value)
- Validate SL-injected headers:
  - `X-SecondLife-Shard` must equal "Production" (reject beta grid)
  - `X-SecondLife-Owner-Key` must match a configured SLPA service account UUID (the script owner)
- Validate the verification code (via Task 02-01's service)
- On success:
  - Update the user record: set sl_avatar_uuid, sl_avatar_name, sl_display_name, sl_username, sl_born_date, sl_payinfo, verified=true, verified_at=now
  - Mark the verification code as used
  - Return success response (the LSL script will display a confirmation message)
- On failure:
  - Return appropriate error code/message (invalid code, expired code, wrong shard, avatar already linked to another account, etc.)
- Enforce one-avatar-per-account: reject if the avatar UUID is already linked to a different user

## Acceptance Criteria

- Endpoint accepts POST with verification data and SL headers
- Requests from non-Production shard are rejected with 403
- Requests with wrong/missing Owner-Key are rejected with 403
- Valid code + valid headers → user account updated with all SL avatar data, marked verified
- Invalid/expired/used code → appropriate error returned
- Attempting to link an avatar already linked to another account → rejected with 409
- Attempting to verify an already-verified account → rejected with 409
- Can be tested with curl by manually setting X-SecondLife-* headers

## Notes

- The X-SecondLife-Owner-Key is the UUID of the avatar who OWNS the script, not the avatar touching the terminal. The touching avatar's data comes in the request body. The owner key should match a configured SLPA service account - this ensures only official SLPA scripts can call this endpoint.
- The SLPA service account UUID should be configurable via application properties.
- Response format should be simple text or JSON that the LSL script can parse and display to the user.
- This endpoint must NOT require JWT auth - it's called from SL, not from a browser.
